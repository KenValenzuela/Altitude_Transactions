# Altitude API Contracts and Mock Boundaries

Date: 2026-05-30

This document describes the frontend-facing FastAPI contracts used by the upgraded workflow. It intentionally separates real persisted behavior from demo/mock behavior so the UI does not imply capabilities that are not implemented yet.

## Real API boundaries in use

| Workflow step | Frontend method | FastAPI route | Purpose | Real vs mocked |
| --- | --- | --- | --- | --- |
| Dashboard queue | `api.listTransactions()` | `GET /api/transactions` | Returns transaction cards with progress, status, price, parties, and next deadline copy. | Real persisted transaction data seeded/generated by backend. |
| Transaction detail | `api.getTransaction(id)` | `GET /api/transactions/{transaction_id}` | Returns workspace data: documents, extraction runs, extracted fields, deadlines, tasks, contacts, requirements, post-close tasks, and audit events. | Real persisted workspace. |
| Upload contract | `api.uploadDocument(file)` | `POST /api/documents/upload` | Stores source document and creates extraction run. | Real upload/job boundary; parser is mocked. |
| Extraction status | `api.getExtraction(documentId)` | `GET /api/documents/{document_id}/extraction` | Returns extraction run, review fields, generated deadlines, and flags. | Real job lookup; field content comes from mocked extraction service. |
| Confirm extraction | `api.confirmExtraction(jobId)` | `POST /api/extractions/{job_id}/confirm` | Materializes transaction workspace from approved extraction. | Real persistence/materialization. |
| Approve/edit field | `api.approveField(id, value?)` | `PATCH /api/extracted-fields/{field_id}` | Approves, edits, or rejects an extracted field and writes audit history. | Real mutation. |
| Update task | `api.updateTask(id, state)` | `PATCH /api/tasks/{task_id}` | Updates task status and writes audit history. | Real mutation. |
| Update document requirement | `api.updateDocument(id, state)` | `PATCH /api/documents/{document_id}` | Updates document requirement received/reviewed status. | Real mutation. |
| Activity log | Included in transaction detail and available via `GET /api/transactions/{id}/audit` | Audit event list. | Real audit rows. |

## Recommended contracts to add next

| Contract | Proposed route | Why |
| --- | --- | --- |
| Dashboard summary | `GET /api/dashboard/summary` or `GET /api/transactions/summary` | Avoid deriving operational dashboard metrics solely in the browser as the portfolio grows. |
| Extraction review PATCH batch | `PATCH /api/extractions/{job_id}/fields` | Approve/edit multiple fields at once and reduce request overhead. |
| Deadline update | `PATCH /api/deadlines/{deadline_id}` | Explicitly support deadline completion/N/A workflows rather than treating deadlines as read-only. |
| Activity stream pagination | `GET /api/transactions/{id}/audit?cursor=` | Keep large transaction histories performant. |

## Response shape principles

- Use camelCase JSON for frontend responses.
- Preserve domain-specific status enums instead of free-form strings.
- Include source/evidence references on extracted values.
- Return the updated object from PATCH routes so the UI can update local state without a full reload.
- Keep mock extraction behind the service boundary only; do not mock API responses in UI components unless explicitly marked as fixtures.

## Current known mock/stub behavior

- AI/PDF parsing is mocked by `MockExtractionService`, which replays structured sample CTME extraction data.
- Auth is stubbed to a seeded broker/session.
- The frontend contains typed fixtures for historical walkthrough/offline fallback use only; production workflow screens call FastAPI.
